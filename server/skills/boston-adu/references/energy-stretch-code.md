# Boston Energy Code

## Boston Has Adopted The Specialized Opt-In Code

Massachusetts offers municipalities three energy-code paths:

1. **Base Code** — IECC 2021 with MA amendments (the statewide default; ~30% of MA municipalities by adoption count)
2. **Stretch Energy Code (225 CMR 22)** — higher performance than Base; the majority of MA municipalities
3. **Specialized Opt-In Code (225 CMR 22 Appendix RC)** — net-zero or near-net-zero performance; ~60 municipalities including Boston, Cambridge, Somerville, Newton, and Brookline

**Practical implication for ADUs**: every new-construction or gut-rehab ADU in Boston is subject to Specialized Opt-In Code requirements. There is no opt-down to Stretch or Base for Boston ADU work. Treat Specialized as the assumed compliance target unless project-specific documentation establishes a carve-out.

## What Specialized Opt-In Requires (Compared To Base / Stretch)

The Specialized Opt-In Code's defining requirement is **electrification or renewable offset**: the dwelling must either heat (and provide hot water) entirely with electricity, or compensate for fossil-fuel use with on-site renewable generation. The compliance pathways are:

### Pathway 1: All-Electric

- Heat pump for space heating and cooling
- Heat pump water heater (or heat-pump-tank or HPWH-supplemented system)
- Electric range / induction (or fossil cooking with documented exemption)
- No fossil-fuel HVAC equipment

### Pathway 2: Mixed-Fuel With Solar PV Offset

- Fossil-fuel space and water heating allowed
- Solar PV system sized to offset the projected fossil-fuel energy consumption (not just electricity)
- Solar-ready provisions standard regardless

### Pathway 3: Passive House Certification (or Equivalent High-Performance)

- Passive House Institute (PHI / Passivhaus) or PHIUS+ certification
- Or equivalent demonstrated performance via energy modeling

## Envelope Requirements

Specialized Opt-In Code envelope R-values exceed Base Code by approximately 15-30%, depending on assembly. Typical Boston ADU envelope targets:

- **Wall**: R-21 cavity + R-5 continuous (effective ~R-26) for 2x6 framing; higher for 2x4
- **Roof / ceiling**: R-49 to R-60
- **Floor / slab edge**: R-19 cavity + R-10 perimeter
- **Window U-factor**: ≤0.27 (Specialized) vs. ≤0.30 (Base)
- **Air leakage**: tested ≤3.0 ACH50 (Specialized) vs. ≤5.0 ACH50 (Base)

## HERS Rating Targets

Boston ADUs under Specialized Opt-In typically target **HERS 45-55** (lower is better; HERS 100 is the IECC 2006 reference home). HERS 45-55 represents ~50% better than the reference home.

[source: c:\dev\permitmonkey\.claude\skills\ma-city-research\references\boston.md (last_verified 2026-04-22) | retrieved: 2026-05-03 | citation: city-research file states "HERS rating typically 45-55"]

**Compliance documentation typically required at permit:**

- HERS rating from a certified RESNET HERS rater
- Energy modeling output (PHIUS or RESNET REM/Rate)
- Solar PV plans (if Pathway 2)
- Heat pump specifications and load calc (if Pathway 1)
- Mechanical ventilation design (HRV/ERV typical for tight envelopes)

## Mechanical Ventilation

The Specialized Opt-In Code's tight envelope requirements effectively force balanced mechanical ventilation. IRC M1505 (referencing ASHRAE 62.2) requires whole-house ventilation in any new dwelling; the Specialized envelope tightness target makes a passive air-leakage approach insufficient.

Practical default for a Boston ADU: **HRV (heat recovery ventilator) or ERV (energy recovery ventilator)** with continuous balanced flow sized per ASHRAE 62.2. A bath-fan-only ventilation strategy is unlikely to satisfy plan review under Specialized.

## Solar-Ready Provisions

Specialized Opt-In Code requires solar-ready provisions even for projects not installing solar PV at construction:

- **Conduit pathway** from main electrical panel to roof
- **Electrical panel capacity** sized for future solar interconnection
- **Structural roof load** sized to support solar PV array weight
- **Roof orientation and shading** documented (south-facing primary roof preferred)

These add modest construction cost and minimal design complexity but are routinely flagged in Boston ADU plan reviews when omitted from drawings.

## Common Boston ADU Energy Corrections

The most frequent Specialized Opt-In Code corrections in Boston ADU plan reviews:

1. **Missing HERS rating documentation** — plan reviewer wants the rater's report attached
2. **Heat pump load calc omitted or undersized** — Manual J calculations required for the heat pump path
3. **HRV/ERV design missing** — Specialized envelope without balanced ventilation triggers correction
4. **Solar-ready provisions not detailed** — drawings need to call out the conduit pathway and panel capacity
5. **Window U-factor cited at 0.30** — that's Base Code; Specialized is 0.27
6. **Air leakage target not specified** — drawings should call out 3.0 ACH50 max
7. **Fossil cooking without mixed-fuel pathway documentation** — if claiming Pathway 1 (all-electric), the cooking range must be electric or induction

For each, the agent's response cites:
- **Authority**: 225 CMR 22 Appendix RC + the specific section
- **Boston application context**: "Boston has adopted the Specialized Opt-In Code; IECC 2021 / Base Code does not apply"
- **Compliance demonstration**: HERS rating, energy model output, calc sheets, manufacturer data

## Workflow For The Agent

When a Boston ADU correction touches energy code:

1. **Confirm Specialized Opt-In is the applicable code**, not Base or Stretch. Quote from this file or `building-codes.md`.
2. **Identify which compliance pathway the project is using** (all-electric / mixed-fuel + solar / Passive House). The pathway determines which specific corrections apply.
3. **Reference IRC Chapter 11 (energy provisions) modified by 225 CMR 22 Appendix RC** for the applicable section. Do NOT cite IECC 2021 base sections without the Specialized overlay.
4. **For envelope corrections**, quote the specific R-value, U-factor, or ACH50 target from current code. Mark TBD if not verified.
5. **For HERS / energy modeling corrections**, frame the response around providing the missing documentation, not arguing the requirement.
